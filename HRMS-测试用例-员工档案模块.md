## 一、测试概述

### 1.1 测试目标

验证 HRMS 员工档案管理模块的功能完整性、业务规则正确性、前后端交互体验及字段权限控制能力，覆盖员工 CRUD（6个接口）、高级搜索、变更历史、敏感数据脱敏等全部后端 API 及前后端联动场景。

### 1.2 测试范围

| 模块 | 子功能 | 接口数 | 说明 |
|---|---|---|---|
| 员工管理 | 员工列表 | 1 | `GET /employees/list`，分页+高级搜索（关键词/部门多选/职位多选/状态多选/职级多选/日期范围） |
| 员工管理 | 员工详情 | 1 | `GET /employees/detail`，四分区（基础/个人/工作/薪资合同） |
| 员工管理 | 新增员工 | 1 | `POST /employees/add`，含工号自动生成+系统账号创建 |
| 员工管理 | 编辑员工 | 1 | `PUT /employees/update`，字段权限控制+变更日志 |
| 员工管理 | 删除员工 | 1 | `POST /employees/delete`，软删除 |
| 员工管理 | 变更历史 | 1 | `GET /employees/change-logs` |

### 1.3 测试环境

| 项目 | 说明 |
|---|---|
| 基础路径 | `http://localhost:8123/api` |
| 接口文档 | `http://localhost:8123/api/doc.html`（Knife4j） |
| 数据库 | MySQL `szml` |
| 前置条件 | `employee`、`employee_detail`、`employee_change_log` 表已建；`department`、`position` 表有初始化数据 |
| 统一响应格式 | `{ "code": 0, "data": ..., "message": "ok" }` |
| 错误码约定 | `code=0` 成功，`code=40000` 参数错误，`code=50001` 操作失败，`code=40400` 资源不存在 |

### 1.4 数据库表

- **employee（主表）**：工号、姓名、性别、手机号、状态、部门、职位、入职日期等核心字段
- **employee_detail（详情表）**：身份证、银行账号、合同、薪资等敏感/补充字段，一对一关联
- **employee_change_log（变更日志表）**：字段变更审计

### 13.3 关注重点

1. **系统账号自动创建**：新增员工时自动创建 user 记录，账号=手机号，初始密码随机8位
2. **敏感数据脱敏**：身份证前4后4、手机前3后4、银行账号仅后4位，需验证脱敏一致性
3. **字段权限控制**：HR/部门主管/普通员工/财务四种角色的字段可见性和可编辑性差异
4. **锁定字段拒绝修改**：手机号/身份证/部门/职位/职级/入职类型等锁定字段后端必须拒绝直接修改
5. **变更历史审计**：编辑后仅记录实际变更的字段，值未变的字段不产生日志
6. **详情四分区 VO 结构**：基础/个人/工作/薪资合同四区嵌套 JSON 结构完整性
7. **高级搜索多选参数**：departmentIds/positionIds/statuses/jobLevels 均为多选，日期范围为区间查询
8. **前后端数据源联动**：新增/编辑页的部门、职位、职级、汇报人下拉数据源分别来自不同接口
9. **编辑页锁定字段提示**：前端 disabled + Tooltip 提示原因，后端兜底拒绝
