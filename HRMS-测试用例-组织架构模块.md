## 一、测试概述

### 1.1 测试目标

验证 HRMS 系统组织架构模块的功能完整性、业务规则正确性、边界容错能力及前后端交互体验，覆盖部门管理（6个接口）和职位管理（5个接口）的全部后端 API，以及前端页面与后端接口的联动场景。

### 1.2 测试范围

| 模块 | 子功能 | 接口数 | 说明 |
|---|---|---|---|
| 部门管理 | 部门树查询 | 1 | `GET /departments/tree`，全量加载+递归人数统计 |
| 部门管理 | 部门详情 | 1 | `GET /departments/detail?id=` |
| 部门管理 | 部门新增 | 1 | `POST /departments/add`，含编码唯一、层级校验、同级名称唯一 |
| 部门管理 | 部门编辑 | 1 | `PUT /departments/update`，编码/上级不可改 |
| 部门管理 | 部门删除 | 1 | `POST /departments/delete`，校验无子部门+无在职员工 |
| 部门管理 | 部门合并 | 1 | `POST /departments/merge`，事务性员工+子部门转移 |
| 职位管理 | 职位列表 | 1 | `GET /positions/list`，支持序列+部门筛选 |
| 职位管理 | 职位新增 | 1 | `POST /positions/add`，含序列/职级/唯一性校验 |
| 职位管理 | 职位编辑 | 1 | `PUT /positions/update` |
| 职位管理 | 职位删除 | 1 | `POST /positions/delete`，校验无员工引用 |
| 职位管理 | 序列职级对照 | 1 | `GET /positions/sequences` |

### 1.3 测试环境

| 项目 | 说明 |
|---|---|
| 基础路径 | `http://localhost:8123/api` |
| 接口文档 | `http://localhost:8123/api/doc.html`（Knife4j） |
| 数据库 | MySQL `szml` |
| 前置条件 | 已执行 `sql/org_structure.sql` 建表及初始化数据 |
| 统一响应格式 | `{ "code": 0, "data": ..., "message": "ok" }` |
| 错误码约定 | `code=0` 成功，`code=40000` 参数错误，`code=50001` 操作失败，`code=40400` 资源不存在 |

### 1.4 初始化数据

```sql
-- 默认根部门
(1, '总公司', '00', NULL, 0, '公司根部门')

-- 默认职位
(1, '总经理', 1, NULL, 'M3', 'M5', 3, '公司总经理，管理序列')
(2, '部门经理', 1, NULL, 'M1', 'M3', 3, '部门经理，管理序列')
(3, '高级开发工程师', 2, NULL, 'P6', 'P8', 3, '高级开发，专业序列')
(4, '开发工程师', 2, NULL, 'P3', 'P6', 3, '初中级开发，专业序列')
(5, '行政专员', 3, NULL, 'S1', 'S3', 3, '行政支持，支持序列')
```

## 九、测试执行建议

### 9.1 执行优先级

1. **P0（冒烟测试）**：87 条，覆盖核心功能路径和关键前后端交互，每次发版必测
2. **P1（回归测试）**：113 条，覆盖主要业务规则、校验逻辑和交互细节
3. **P2（完整测试）**：82 条，覆盖边界、异常、并发、UI 细节等场景

### 9.2 测试数据准备

- 使用 `sql/org_structure.sql` 初始化表和基础数据
- 使用 `sql/employee_test_data.sql` 准备员工测试数据
- 每个测试场景前确认数据隔离，避免用例间相互干扰
- 合并/删除等破坏性操作建议在独立事务中或使用独立测试数据集

### 9.3 关注重点

1. **部门合并事务**：涉及员工批量转移+子部门转移+源部门删除+日志记录，需验证事务一致性
2. **编码不可回收**：已删除部门的编码不可复用，需验证 `countByCodeIgnoreDelete` 绕过 `@TableLogic` 的正确性
3. **层级深度校验**：`calculateDepth` 递归向上计数，需验证5级边界的准确性
4. **人数递归统计**：`buildEmployeeCountMap` 需确保子孙部门ID收集完整
5. **职位唯一性**：同序列+名称+部门（含null处理）的唯一性校验
6. **前端校验与后端兜底双重验证**：前端拦截同名重复、必填项等，后端必须同样校验，防止绕过前端直接调用
7. **弹窗提交后数据刷新**：新增/编辑/删除/合并成功后，部门树和职位列表是否自动刷新，右侧面板是否联动更新
8. **序列-职级级联联动**：前端切换序列后职级下拉选项是否正确联动，编辑时序列变更是否清空旧职级值
9. **Loading 和重复提交防护**：所有写操作是否显示 loading、按钮禁用，防止重复创建/删除
10. **接口错误消息透传**：后端返回的业务错误 message 是否被前端正确解析并展示给用户
