# 第一部分：项目概述

## 1. 项目背景

目前公司内部人力资源管理依赖 Excel 和纸质流程，存在数据分散、算薪效率低、审批不透明等问题。本系统旨在建立统一的员工数字化档案，实现入转调离全流程线上化。

## 2. 目标用户及定位

| 用户角色 | 核心需求 | 使用频率 |
|----------|----------|----------|
| **系统管理员** | 全局配置、用户管理、角色分配 | 低频 |
| **HR 专员** | 员工档案管理、薪资核算、考勤统计 | 每日 |
| **部门主管** | 团队人员查看、下属请假审批 | 每日 |
| **普通员工** | 个人信息、请假申请、工资条查看 | 每周 |
| **财务专员** | 薪资审核、成本报表 | 每月 |

## 3. 名词定义

| 名词 | 定义 |
|------|------|
| **在职状态** | 员工当前工作状态：试用期 / 正式 / 待离职 / 已离职 |
| **账套** | 薪资计算模板，定义工资构成项目（基本、津贴、扣除等） |
| **考勤组** | 适用相同考勤规则的员工集合 |
| **应发工资** | 税前应发 = 基本 + 津贴 + 绩效 + 加班 |
| **实发工资** | 税后实发 = 应发 - 社保 - 公积金 - 个税 |
| **入转调离** | 入职、转正、调岗、离职四大人事异动 |

## 4. 系统布局

- **左侧**：可折叠多级菜单导航栏，按功能模块分组，**菜单按角色动态渲染**
- **顶部**：系统 Logo + 全局搜索 + 消息通知 + 用户头像下拉菜单（个人信息/修改密码/退出登录）
- **主体**：面包屑导航 + 页面内容区
- **登录/注册页**：独立于主布局之外，居中卡片式设计

│  salary_account ──1:N── salary_item       账套 → 项目       │

└──────────────────────────────────────────────────────────────┘

## 3. 设计决策

| 决策 | 原因 |
|------|------|
| employee 独立于 user | user 管登录认证，employee 管 HR 敏感字段，字段级权限可控 |
| 考勤+请假+加班四合一 | 避免 3 张表 JOIN，类型字段区分 |
| 审批流通用化 | 6 类业务共用一个审批引擎 |
| 入转调离不建独立表 | 状态存 employee，审批记录存 approval_flow |
| 工资条用 JSON 存项目 | 避免多层 JOIN，读取一次即可渲染 |
| 部门 ID 冗余到考勤/审批 | 减少 JOIN，加速部门维度统计 |

> 完整建表 SQL 见 `sql/hrms_all_tables.sql`

# 第四部分：登录与认证模块

## 1. 功能概述

| 功能 | 说明 |
|------|------|
| 用户注册 | 账号密码注册、账号唯一性校验（synchronized 加锁）、MD5+固定盐值加密 |
| 用户登录 | 账号密码校验 → Session 写入登录态 → 返回脱敏 LoginUserVO |
| 获取当前用户 | 从 Session 读取 → 数据库二次验证 → 返回脱敏信息 |
| 用户注销 | 清除 Session 中的 `user_login` |
| 权限校验 | AOP 切面拦截 `@AuthCheck` 注解 → 基于角色的权限模型 |
| 用户管理 | 管理员创建/删除/更新/分页查询用户 |
| 个人信息修改 | 仅允许修改昵称/头像/简介 |

## 2. 认证流程

```
用户访问 → 是否已登录？
  ├── 已登录 → GET /user/get/login → 返回 LoginUserVO
  └── 未登录 → 登录页 → POST /user/login
                 → 密码 MD5+SALT 加密 → 数据库匹配
                 → Session.setAttribute("user_login", user)
                 → 返回 LoginUserVO（脱敏）
```

## 3. 权限校验（AOP）

`@AuthCheck(mustRole = "xxx")` 注解 → `AuthInterceptor` 切面拦截 → 从 Session 获取用户角色 → 比较权限 → 放行/403。

**角色枚举：** `sys_admin` | `hr` | `dept_manager` | `finance` | `employee` | `ban`

**密码加密：** `MD5("limou" + 明文密码)`

**Session 配置：** 超时 30 天，Cookie `JSESSIONID`，Axios `withCredentials: true`

## 4. 前端页面

| 页面 | 路由 | 说明 |
|------|------|------|
| 登录页 | `/user/login` | 居中卡片：账号+密码+记住我+登录按钮 |
| 注册页 | `/user/register` | 居中卡片：账号+密码+确认密码+注册按钮 |
| 用户管理页 | `/settings/users` | 管理员专属：分页表格+搜索+新增/编辑/删除 |

## 5. 关键 API

| 方法 | 路径 | 权限 | 说明 |
|------|------|------|------|
| POST | `/api/user/register` | 无 | 注册，密码≥8位，账号≥4位 |
| POST | `/api/user/login` | 无 | 登录，返回 LoginUserVO |
| GET | `/api/user/get/login` | 无 | 获取当前用户（Session） |
| POST | `/api/user/logout` | 无 | 注销 |
| POST | `/api/user/add` | sys_admin | 创建用户，默认密码12345678 |
| POST | `/api/user/delete` | sys_admin | 物理删除 |
| POST | `/api/user/update` | sys_admin | 更新用户信息/角色 |
| POST | `/api/user/list/page` | sys_admin | 分页查询 |
| POST | `/api/user/update/my` | 登录即可 | 修改个人信息 |

> 详细系分见：[后端系分-登录模块.md](./后端系分-登录模块.md)、[前端系分-登录模块.md](./前端系分-登录模块.md)

# 第五部分：组织架构管理

## 1. 部门管理

### 1.1 功能定位
维护公司多级组织架构，支持部门增删改查及人员统计。最大层级深度：**5 级**。

### 1.2 部门字段

| 字段 | 必填 | 说明 |
|------|------|------|
| 部门名称 | 是 | 如"技术部" |
| 部门编码 | 是 | 纯数字，如"01"、"0101"，用于工号生成 |
| 上级部门 | 否 | NULL 表示根部门 |
| 部门负责人 | 否 | 决定数据权限 |
| 排序序号 | 是 | 越小越靠前，同级不可重复 |
| 部门描述 | 否 | 职能说明 |

### 1.3 关键规则
- **层级限制**：最大 5 级，创建/移动时校验
- **删除前置校验**：① 是否有子部门 ② 是否有在职员工
- **人数统计**：实时递归汇总（含下属部门），统计口径：试用期 + 正式员工
- **部门编码**：用于工号生成（格式：年份4位 + 部门编码2位 + 序号3位）

### 1.4 关键 API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/v1/departments/tree` | 部门树（含人数统计） |
| GET | `/api/v1/departments` | 平铺列表（分页+搜索） |
| POST | `/api/v1/departments` | 创建部门 |
| PUT | `/api/v1/departments/{id}` | 更新部门 |
| DELETE | `/api/v1/departments/{id}` | 删除（逻辑删除） |

# 第七部分：入转调离流程

## 1. 入职流程

### 1.1 状态流转

```
新建候选人 → 信息录入 → 草稿 → 提交审批 → 审批中
                                            ├── 审批通过 → 已批准待入职 → 已入职
                                            ├── 审批拒绝 → 已拒绝（可重新发起）
                                            └── 转交他人
```

### 1.2 审批规则
- 一级审批：部门负责人（固定）
- 二级审批：HR负责人（可配置开关；非标准职位或薪资超出职级范围时自动开启）

### 1.3 通过后自动处理
1. 生成工号（年份+部门编码+序号）
2. 创建系统账号（登录账号=手机号，初始密码随机，首次登录强制修改）
3. 写入员工档案（employee + personal_info + work_info 共3张表）
4. 发送欢迎邮件给候选人
5. 发送确认通知给 HR 和部门负责人

## 2. 转正流程

```
试用期 → 系统提前7天提醒HR → HR发起转正评估 → 转正审批（部门负责人→HR负责人）
                                                    ├── 通过 → 已转正（状态变"正式"）
                                                    └── 不通过 → 延长试用 or 辞退
```

## 3. 调岗流程

```
HR发起 → 原部门负责人审批（知情确认）→ 新部门负责人审批（接收确认）
       → HR负责人备案（最终确认）→ 调岗生效
```

**生效处理**：更新部门/职位/职级/汇报人；工号不变；记录调岗历史。

## 4. 离职流程

```
HR发起 → 部门负责人审批（确认交接）→ HR负责人审批 → 审批通过
       → 待离职（等待最后工作日）→ 已离职
```

**生效处理**：状态变更为已离职 → 账号禁用 → 从考勤组移除 → 薪资计算至离职日期 → 档案保留可查询。

## 5. 关键 API

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/v1/onboarding` | 创建入职申请 |
| POST | `/api/v1/onboarding/{id}/submit` | 提交审批 |
| POST | `/api/v1/regularization/{employeeId}` | 发起转正 |
| POST | `/api/v1/transfer/{employeeId}` | 发起调岗 |
| POST | `/api/v1/resignation/{employeeId}` | 发起离职 |

> 审批流转统一走审批中心（见第十部分）

> 详细系分见：[入转调离-后端系分.md](../doc-s/入转调离-后端系分.md)

## 2. 员工薪资档案

每个员工独立维护：适用账套 + 基本工资 + 津贴基数 + 社保基数 + 公积金基数 + 绩效基数。

**更新自动记录调薪历史**（change_type 自动识别：1=调薪 2=账套变更 3=基数调整 4=转正调薪 5=调岗调薪）。

**试用期薪资**：按试用期比例（80%-100%）计算，仅影响基本工资和津贴。

## 3. 月度薪资核算

### 3.1 核算流程

```
新建核算批次（选择月份）
  → 数据准备（考勤+请假+加班+薪资档案）
  → 异步自动计算（@Async 线程池，策略模式计算引擎）
    → 遍历员工 → 取档案 → 考勤扣款 → 应用账套公式 → 累计预扣法个税
  → 核算预览（分页表格+异常检测+手动调整）
  → 提交审批 → 审批流转 → 审批通过 → 工资条可见 → 标记已发放
```

### 3.2 批次状态流转

```
草稿(0) → 计算中(1) → 待确认(2) → 审批中(3) → 已通过(4) → 已发放(5)
                                                ↘ 已驳回(6) → 回到待确认(2)
```

### 3.3 异常检测规则

| 条件 | 级别 | is_abnormal |
|------|------|-------------|
| 当月请假 > 15天 | 黄色预警 | 1 |
| 当月加班 > 50小时 | 黄色预警 | 1 |
| 薪资较上月变动 > 30% | 红色预警 | 2 |
| 新入职未设薪资档案 | 红色阻断 | 3 |

### 3.4 个税累计预扣法

```
累计应纳税所得额 = 累计收入 - 累计起征点(5000×N) - 累计社保 - 累计公积金 - 累计专项附加扣除
累计应纳税额 = 累计应纳税所得额 × 税率 - 速算扣除数
当月个税 = max(累计应纳税额 - 累计已缴税额, 0)
```

**7 级累进税率：** 3%（≤36,000）→ 10% → 20% → 25% → 30% → 35% → 45%（>960,000）

**计算常量：** 月计薪天数 21.75 | 日工作小时 8 | 月个税起征点 5000

### 3.5 计算引擎（策略模式）

```
SalaryCalculatorEngine
  ├── FixedIncomeCalculator      → 基本工资 + 津贴 × 试用期比例
  ├── VariableIncomeCalculator   → 绩效基数 × 系数 + 小时工资 × 加班倍数 × 小时
  ├── AttendanceDeductionCalculator → -(50×迟到次数 + 日工资×请假天数)
  ├── SocialSecurityCalculator   → -(基数 × 10.5%)
  ├── HousingFundCalculator      → -(基数 × 12%)
  └── IncomeTaxCalculator        → 累计预扣法
```

所有金额使用 **BigDecimal**，保留 2 位小数，舍入模式 `ROUND_HALF_UP`。

## 4. 工资条（员工自助）

# 第十三部分：技术架构

## 1. 技术选型

| 类别 | 技术 | 说明 |
|------|------|------|
| 后端框架 | Spring Boot | — |
| ORM | MyBatis-Plus | BaseMapper + QueryWrapper |
| 数据库 | MySQL 8.0 | InnoDB, utf8mb4 |
| 缓存 | Redis 7.x | Session 共享、验证码、待办缓存 |
| 消息队列 | RabbitMQ | 审批通知、状态同步、操作审计 |
| 接口文档 | Swagger/Knife4j | `/api/v2/api-docs` |
| 前端框架 | UmiJS Max 4 + Ant Design Pro 5 | React |
| 图表库 | AntV | 薪资统计 + 考勤统计可视化 |

## 2. 认证方案

- **Session-Cookie** 认证（JSESSIONID）
- Session 超时：30 天
- 密码加密：MD5 + 固定盐值（生产建议升级为 BCrypt）
- 权限控制：AOP 切面 + @AuthCheck 注解

## 3. 异步计算

- 薪资核算使用 `@Async("salaryTaskExecutor")` 线程池
- 核心 4 线程，最大 8 线程，队列 100
- 前端轮询批次状态（3 秒间隔，5 分钟超时）

## 4. 数据权限实现

- `DataScopeContext`：请求级别 Bean，存储当前用户角色和部门范围
- `DataScopeHelper`：对 MyBatis-Plus QueryWrapper 动态追加 WHERE 条件
- 部门主管：递归查询子部门 ID → `department_id IN (...)`
- 普通员工：`employee_id = 当前用户ID`
