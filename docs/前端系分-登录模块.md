## 1. 概述

### 1.1 项目背景

目前公司内部人力资源管理依赖 Excel 和纸质流程，存在数据分散、算薪效率低、审批不透明等问题。本系统旨在建立统一的员工数字化档案，实现入转调离全流程线上化。

本次迭代为 HRMS 系统**登录模块**的完整前端实现，涵盖用户登录页、注册页、登录态管理、角色权限体系（路由守卫 + 按钮指令）、退出登录等核心功能。登录模块是系统的入口模块，所有其他业务页面的访问都依赖于登录模块的正确运行。

### 1.2 目标用户及定位

| 用户角色 | 核心需求 | 使用频率 |
|----------|----------|----------|
| **系统管理员** | 全局配置、用户管理、角色分配 | 低频 |
| **HR 专员** | 员工档案管理、薪资核算、考勤统计 | 每日 |
| **部门主管** | 团队人员查看、下属请假审批 | 每日 |
| **普通员工** | 个人信息、请假申请、工资条查看 | 每周 |
| **财务专员** | 薪资审核、成本报表 | 每月 |

### 1.3 名词定义

| 名词 | 定义 |
|------|------|
| **登录态** | 基于 Session-Cookie，浏览器自动携带 `JSESSIONID` |
| **脱敏** | `LoginUserVO` 仅返回 id/userName/userAvatar/userProfile/userRole，不包含密码等敏感字段 |
| **角色层级** | `sys_admin(100) > hr(80) > dept_manager(60) > finance(40) > employee(20) > ban(0)`，高层级自动继承低层级权限 |

### 1.4 后台管理系统的布局

- **左侧**：可折叠多级菜单导航栏，按功能模块分组，**菜单按角色动态渲染**
- **顶部**：系统 Logo + 全局搜索 + 消息通知 + 用户头像下拉菜单（个人信息/修改密码/退出登录）
- **主体**：面包屑导航 + 页面内容区
- **登录页**：独立于主布局之外，居中卡片式设计

### 2.4 前端登录与权限实现方案

#### 2.4.1 登录流程

```
用户访问系统 → 是否已登录？
  ├── 已登录 → 获取用户信息 GET /api/user/get/login
  │              → 返回 LoginUserVO(id, userName, userRole...)
  │              → 存储到 Pinia/Vuex Store
  │              → 根据 userRole 拉取菜单权限、渲染菜单
  │              → 路由守卫校验页面访问权限
  └── 未登录 → 跳转登录页 /login
                 → 输入账号密码 → POST /api/user/login
                 → 成功后存储用户信息 → 跳转首页
```

#### 2.4.2 登录页 UI

- **页面路由**：`/login`
- **布局**：居中卡片式登录框，独立于主布局（无左侧菜单和顶部导航）
- **最终渲染**：一个居中卡片表单

| 字段名称 | 输入方式 | 是否必填 | 校验规则 | 提示文案 |
|----------|----------|----------|----------|----------|
| 账号 | Input | Y | 非空，最少 4 位 | 请输入账号 |
| 密码 | Input（可切换明文/密文）| Y | 非空，最少 8 位 | 请输入密码 |
| 记住我 | Checkbox | N | — | 记住登录状态（7天）|

| 按钮 | 交互 |
|------|------|
| 登录 | 校验 → 调用 POST `/api/user/login` → 存储用户信息 → 跳转首页 `/` |
| 忘记密码 | 弹出提示"请联系管理员重置密码" |

**异常状态**：
- 账号或密码错误 → "账号或密码错误，请重试"
- 网络超时 → "网络连接失败，请检查网络后重试"
- 账号被禁用 → "账号已被禁用，请联系管理员"
- 参数为空 → "请输入账号和密码"

#### 2.4.3 注册页 UI

- **页面路由**：`/register`
- **布局**：居中卡片式，与登录页风格一致

| 字段名称 | 输入方式 | 是否必填 | 校验规则 | 提示文案 |
|----------|----------|----------|----------|----------|
| 账号 | Input | Y | 最少 4 位 | 请输入账号（至少4位）|
| 密码 | Input（可切换明文/密文）| Y | 最少 8 位 | 请输入密码（至少8位）|
| 确认密码 | Input | Y | 与密码一致 | 请再次输入密码 |

| 按钮 | 交互 |
|------|------|
| 注册 | 校验 → 调用 POST `/api/user/register` → 成功后跳转登录页 |
| 返回登录 | 跳转 `/login` |

**异常状态**：
- 密码不一致 → "两次输入的密码不一致"
- 账号已存在 → "该账号已被注册"
- 注册成功 → "注册成功，请登录" → 跳转 `/login`

#### 2.4.4 Token/Session 管理

- **方案**：基于 Session-Cookie，Axios 配置 `withCredentials: true`
- **Session 过期**（默认 30 分钟，可通过 `server.servlet.session.timeout` 配置延长）：后端返回 401 → 拦截器统一处理 → 清除 Store → 跳转登录页
- **退出登录**：调用 POST `/api/user/logout` → 清除 Store → 跳转登录页
- **页面刷新恢复**：App 挂载时调用 GET `/api/user/get/login` → 成功则恢复登录态 → 失败则跳转登录页

#### 2.4.5 路由权限控制

```javascript
// 路由配置示例
const routes = [
  {
    path: '/salary/account',
    component: () => import('@/views/salary/AccountList.vue'),
    meta: { title: '薪资账套', role: ['sys_admin', 'hr'] }
  },
  {
    path: '/salary/batch',
    component: () => import('@/views/salary/BatchList.vue'),
    meta: { title: '薪资核算', role: ['sys_admin', 'hr', 'finance'] }
  },
  {
    path: '/salary/payslip',
    component: () => import('@/views/salary/PayslipList.vue'),
    meta: { title: '我的工资条', role: ['sys_admin', 'hr', 'dept_manager', 'finance', 'employee'] }
  },
  {
    path: '/system/user',
    component: () => import('@/views/system/UserManage.vue'),
    meta: { title: '用户管理', role: ['sys_admin'] }
  }
]

// 路由守卫
router.beforeEach((to, from, next) => {
  const userStore = useUserStore()
  const userRole = userStore.userInfo?.userRole

  // 访问登录页/注册页：已登录则跳首页，未登录则放行
  if (to.name === 'Login' || to.name === 'Register') {
    return userRole ? next({ name: 'Home' }) : next()
  }

  // 其他页面：未登录跳登录页
  if (!userRole) {
    return next({ name: 'Login', query: { redirect: to.fullPath } })
  }

  // 检查页面级权限
  const requiredRoles = to.meta.role
  if (requiredRoles && !requiredRoles.includes(userRole)) {
    return next({ name: 'Forbidden' })
  }

  next()
})
```

#### 2.4.6 按钮/元素级权限指令

```html
<!-- 角色级别控制 -->
<el-button v-permission:role="['sys_admin', 'hr']">新建账套</el-button>
<el-button v-permission:role="['sys_admin']">用户管理</el-button>

<!-- 权限码级别控制（配合后端 @AuthCheck） -->
<el-button v-permission:code="['salary:batch:approve']">审批通过</el-button>
<el-button v-permission:code="['user:delete']">删除用户</el-button>

<!-- 禁用模式：显示但不可点击 -->
<el-button v-permission:code.disable="['salary:account:delete']">删除</el-button>
```

**指令实现逻辑**：

```javascript
// v-permission:role 指令
app.directive('permission', {
  mounted(el, binding) {
    const userStore = useUserStore()
    const userRole = userStore.userInfo?.userRole

    if (binding.arg === 'role') {
      const allowedRoles = binding.value
      if (!allowedRoles.includes(userRole)) {
        el.parentNode?.removeChild(el)  // 移除 DOM
      }
    }

    if (binding.arg === 'code') {
      const requiredCodes = binding.value
      const userPermissions = userStore.permissions
      const hasPermission = requiredCodes.some(code => userPermissions.includes(code))

      if (binding.modifiers?.disable) {
        // 禁用模式：保留 DOM 但置灰 + tooltip
        if (!hasPermission) {
          el.setAttribute('disabled', 'true')
          el.classList.add('is-disabled')
        }
      } else {
        // 隐藏模式：直接移除 DOM
        if (!hasPermission) {
          el.parentNode?.removeChild(el)
        }
      }
    }
  }
})
```

#### 2.4.7 前端 Store 状态设计

```typescript
// stores/user.ts
interface UserState {
  userInfo: {
    id: number
    userName: string
    userAccount: string
    userRole: UserRole  // 'sys_admin' | 'hr' | 'dept_manager' | 'finance' | 'employee'
    userAvatar: string
  } | null
  permissions: string[]      // 权限码列表（如需要细粒度控制）
  menus: MenuItem[]          // 当前角色可见的菜单
  isLoggedIn: boolean
}

// 角色层级辅助
const ROLE_LEVEL: Record<string, number> = {
  sys_admin: 100,
  hr: 80,
  dept_manager: 60,
  finance: 40,
  employee: 20,
  ban: 0
}

// 判断当前用户是否有某个角色的权限（层级比较）
function hasRole(userRole: string, requiredRole: string): boolean {
  return (ROLE_LEVEL[userRole] || 0) >= (ROLE_LEVEL[requiredRole] || 0)
}
```

## 3. 详细设计

## 3.1 前端迭代目标

1. **登录页**：账号密码登录 + 表单校验 + 错误提示 + 登录成功跳转
2. **注册页**：账号密码注册 + 两次密码一致校验 + 注册成功跳转登录
3. **登录态管理**：Pinia Store 存储用户信息 + Axios 拦截器 + 页面刷新恢复
4. **权限路由守卫**：未登录拦截 → 登录页；无权限页面 → 403
5. **权限指令**：`v-permission:role` 控制 DOM 显隐 / `v-permission:code` 控制按钮权限
6. **菜单权限**：左侧菜单按角色动态渲染
7. **退出登录**：清除登录态 + 跳转登录页
8. **用户管理页**（管理员）：用户列表 + 新增用户 + 编辑角色 + 删除用户

### 3.2.3 全局登录态管理

#### Axios 拦截器配置

```javascript
// 请求拦截器：自动携带 Cookie
axios.defaults.withCredentials = true

// 响应拦截器：统一处理 401/403
axios.interceptors.response.use(
  response => response,
  error => {
    if (error.response?.status === 401) {
      // 未登录或 Session 过期
      const userStore = useUserStore()
      userStore.clearUser()
      router.push({ name: 'Login', query: { redirect: router.currentRoute.value.fullPath } })
    }
    if (error.response?.status === 403) {
      router.push({ name: 'Forbidden' })
    }
    return Promise.reject(error)
  }
)
```

#### App 启动恢复登录态

```javascript
// App.vue onMounted
async function restoreLoginState() {
  try {
    const res = await userApi.getLoginUser()  // GET /api/user/get/login
    userStore.setUser(res.data)
  } catch {
    // 未登录，不做处理（路由守卫会处理跳转）
  }
}
```

#### 三种状态处理

| 状态 | 展示 |
|------|------|
| **恢复中** | 全局 Loading 遮罩，不渲染任何页面内容 |
| **恢复成功** | 正常渲染页面，Store 中有 userInfo |
| **恢复失败** | 路由守卫拦截 → 跳转登录页 |

## 4. 接口设计（全 10 个 API）

### 通用说明

| 项目 | 说明 |
|------|------|
| 基础路径 | `/api` |
| 认证方式 | Session-Cookie（`JSESSIONID`），Axios 需配置 `withCredentials: true` |
| 响应格式 | `{"code": 0, "message": "success", "data": {...}}` |
| 成功判定 | `code = 0` |

### 1. 用户注册

```

```

| 参数 | 类型 | 必填 | 描述 |
|------|------|------|------|

| userPassword | String | 是 | 密码，≥8 位 |
