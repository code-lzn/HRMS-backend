## 1. 概述

### 1.1 项目背景

目前公司内部人力资源管理依赖 Excel 和纸质流程，存在数据分散、算薪效率低、审批不透明等问题。本系统旨在建立统一的员工数字化档案，实现入转调离全流程线上化。

本次迭代为 HRMS 系统**薪资管理模块**的完整前端实现，涵盖薪资账套配置、员工薪资档案维护、月度薪资核算流转、工资条自助查询、以及 AntV 数据可视化看板。

### 1.2 目标用户及定位

| 用户角色 | 核心需求 | 使用频率 |
|----------|----------|----------|
| **HR 专员** | 员工档案管理、薪资核算、考勤统计 | 每日 |
| **部门主管** | 团队人员情况查看、下属请假审批 | 每日 |
| **普通员工** | 查看个人信息、提交请假、查看工资条 | 每周 |
| **财务专员** | 薪资审核、成本报表 | 每月 |
| **系统管理员** | 权限配置、系统维护 | 低频 |

### 1.3 名词定义

| 名词 | 定义 |
|------|------|
| **在职状态** | 员工当前工作状态：试用期 / 正式 / 待离职 / 已离职 |
| **账套** | 薪资计算模板，定义工资构成项目（基本、津贴、扣除等）|
| **应发工资** | 税前应发 = 基本 + 津贴 + 绩效 + 加班 |
| **实发工资** | 税后实发 = 应发 - 社保 - 公积金 - 个税 |
| **入转调离** | 入职、转正、调岗、离职四大人事异动 |

### 1.4 后台管理系统的布局

- **左侧**：可折叠多级菜单导航栏，按功能模块分组
- **顶部**：系统 Logo + 全局搜索 + 消息通知 + 用户头像下拉菜单（个人信息/修改密码/退出登录）
- **主体**：面包屑导航 + 页面内容区
- **移动端**（工资条模块）：适配手机竖屏，卡片式布局

## 2. 权限体系

### 2.1 角色定义

| 角色 | 角色标识 | 数据范围 | 核心功能 |
|------|----------|----------|----------|
| **系统管理员** | `admin` | 全平台 | 系统配置、角色管理、数据备份 |
| **HR 专员** | `hr` | 全部员工 | 员工管理、薪资核算、考勤管理、审批管理 |
| **部门主管** | `manager` | 本部门及下属 | 本部门员工查看、下属请假/转正/调岗/离职审批 |
| **财务专员** | `finance` | 薪资相关 | 薪资审核、成本报表查看 |
| **普通员工** | `user` | 仅本人 | 个人信息、请假申请、工资条查看 |

> **后端现状**：当前 `UserRoleEnum` 仅定义了 `user` / `admin` / `ban` 三种角色。需与后端对齐，扩展角色枚举以支持 `hr` / `manager` / `finance`。

| 字段 | HR 专员 | 部门主管 | 普通员工 |

### 2.4 前端登录与权限实现方案

#### 2.4.1 登录流程

```
用户访问系统 → 是否已登录？
  ├── 已登录 → 获取用户信息 GET /api/user/get/login
  │              → 返回 LoginUserVO(id, userName, userRole...)
  │              → 存储到 Pinia/Vuex Store
  │              → 根据 userRole 拉取菜单权限、渲染菜单
  │              → 路由守卫校验页面访问权限
  └── 未登录 → 跳转登录页 /login
                 → 输入账号密码 → POST /api/user/login
                 → 成功后存储用户信息 → 跳转首页
```

#### 2.4.2 登录页 UI

- **页面路由**：`/login`
- **布局**：居中卡片式登录框
- **最终渲染**：一个居中卡片表单

| 字段名称 | 输入方式 | 是否必填 | 校验规则 | 提示文案 |
|----------|----------|----------|----------|----------|
| 账号 | Input | Y | 非空 | 请输入账号 |
| 密码 | Input（可切换明文/密文）| Y | 非空，最少6位 | 请输入密码 |
| 记住我 | Checkbox | N | — | 记住登录状态（7天）|

| 按钮 | 交互 |
|------|------|
| 登录 | 校验 → 调用 POST `/api/user/login` → 存储用户信息 → 跳转首页 `/` |
| 忘记密码 | 弹出提示联系管理员 |

**异常状态**：
- 账号或密码错误 → "账号或密码错误，请重试"
- 网络超时 → "网络连接失败，请检查网络后重试"
- 账号被禁用 → "账号已被禁用，请联系管理员"

#### 2.4.3 Token/Session 管理

- **方案**：基于 Session-Cookie，Axios 配置 `withCredentials: true`
- **Session 过期**（30天）：后端返回 401 → 拦截器统一处理 → 清除 Store → 跳转登录页
- **退出登录**：调用 POST `/api/user/logout` → 清除 Store → 跳转登录页

#### 2.4.4 路由权限控制

```javascript
router.beforeEach((to, from, next) => {
  const userRole = store.getters.userRole
  const requiredRole = to.meta.role

  if (!userRole) {
    return to.name === 'Login' ? next() : next({ name: 'Login' })
  }

  if (requiredRole && !hasRole(userRole, requiredRole)) {
    return next({ name: 'Forbidden' })
  }

  next()
})
```

#### 2.4.5 按钮/元素级权限指令

```html
<el-button v-permission:role="['admin', 'hr']">新建账套</el-button>
<el-button v-permission:code="['salary:batch:approve']">审批通过</el-button>
<el-button v-permission:code.disable="['salary:account:delete']">删除</el-button>
```


#### 2.4.7 登录及权限 API

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 用户登录 | POST | `/api/user/login` | 返回 LoginUserVO（含角色）|

## 3. 详细设计

## 3.1 前端迭代目标

1. **登录/注册页**：账号密码登录 + 登录态管理 + 退出
2. **系统布局框架**：左侧菜单 + 顶部导航 + 面包屑 + 主体内容区，菜单按角色动态渲染
3. **薪资账套管理**：列表页 + 新增/编辑表单弹窗 + 工资项目管理（可拖拽排序）
4. **员工薪资档案**：详情页 + 编辑表单 + 调薪历史
5. **月度薪资核算**：批次列表页 + 核算预览页 + 异常处理 + 审批流转
6. **工资条（员工自助）**：列表 + 详情（含二次验证）+ 个人薪资趋势
7. **薪资数据可视化**：AntV 图表看板（5种图表）
8. **403/404 错误页面**

#### 工资项目子表字段

| 字段名称 | 输入方式 | 是否必填 | 默认值 | 字段类型 | 校验规则 | 数据源 |
|----------|----------|----------|--------|----------|----------|--------|
| 项目名称 | Input | Y | N | string | 非空，1-30字符 | 如：基本工资、绩效奖金 |
| 项目类型 | Select | Y | N | number | 必选 | 枚举：1=固定收入 2=变动收入 3=考勤扣款 4=社保扣除 5=公积金扣除 6=个税 |
| 计算公式 | Input | N | N | string | 最多200字符 | 如：base_salary * 1.0 |
| 排序序号 | InputNumber | Y | 自动递增 | number | 正整数 | — |
| 是否计税 | Switch | Y | true | boolean | — | — |

**子表交互**：
- **添加一行**：在子表末尾新增空白行，排序序号自动 = 最大序号 + 1
- **删除一行**：删除该行；编辑已有账套时，已持久化的项目调 DELETE 接口，新添加未保存的直接从列表移除
- **拖拽排序**：鼠标按住行首拖拽图标 ≡ 上下拖动，松手后自动更新所有行的 sort_order
- **保存时机**：编辑已有账套时，新增/删除项目即时调 API；新建时，所有项目随账套一起提交

#### 操作按钮

| 字段名称 | 交互 | 二次确认 |
|----------|------|----------|
| 保存 | 校验所有字段 → 调 POST/PUT API → 成功后关闭 Modal → 列表刷新 | N |
| 取消 | 关闭 Modal，丢弃所有修改 | 表单有未保存内容时弹窗"确认放弃？" |

#### 表单状态处理

| 状态 | 展示 |
|------|------|
| **编辑-加载中** | Modal 内展示 Spin，表单字段置灰不可编辑 |
| **提交中** | 保存按钮 loading 转圈，不允许重复点击 |
| **校验失败** | 对应字段下方显示红色错误文案，自动滚动到第一个错误字段 |
| **提交失败** | alert 弹窗或 toast 提示错误信息，Modal 不关闭，表单数据保留 |

### 3.2.10 接口联调测试用例（全 21 个 API）

以下测试用例覆盖所有后端接口的**正常流程、边界值、异常状态、权限校验**，前端联调和 QA 测试时逐条执行。

#### 一、薪资账套管理（6 个接口）

##### 1. GET 查询账套列表

| 编号 | 测试场景 | 操作 | 预期结果 |
|------|---------|------|---------|
| API-01-01 | 正常查询 | 不带参数调 GET `/salary-accounts` | code=0，返回全部账套，data 为数组 |
| API-01-02 | 名称模糊搜索 | `?name=标准` | 只返回名称包含"标准"的账套 |
| API-01-03 | 搜不存在的名称 | `?name=不存在的账套` | code=0，data=[]，不报错 |
| API-01-04 | 按 scope_type 筛选 | `?scope_type=1` | 只返回 scope_type=1（部门）的账套 |
| API-01-05 | scope_type 非法值 | `?scope_type=99` | code=0，data=[]（无匹配）或 code≠0 参数校验失败 |
| API-01-06 | 组合筛选 | `?name=标准&scope_type=1` | 两个条件同时生效 |
| API-01-07 | 返回数据包含 items | 查看响应 JSON | 每个账套的 items 字段为数组（可能为空）|

## 7. 其他

### 7.1 风险评估

| 风险项 | 风险等级 | 影响范围 | 应对措施 |
|--------|----------|----------|----------|
| 薪资计算异步任务超时 | 中 | 核算流程中断 | 前端轮询5分钟超时 → 提示手动刷新；后端幂等可重试 |
| 二次验证短信不可用 | 低 | 员工无法查看工资条 | 前端降级为密码验证 |
| 大数据量薪资预览表格卡顿 | 中 | 预览页性能 | 后端分页 + 前端虚拟滚动（vue-virtual-scroller/react-window）|
| AntV 图表大数据量渲染慢 | 低 | 统计看板体验 | 后端聚合数据；部门过多时 TopN + "其他"合并 |
| 权限过滤遗漏 | 高 | 薪资数据泄露 | 路由守卫 + v-permission指令 + 后端接口过滤三重保障 |
| Session 过期频繁登录 | 低 | 用户体验 | 30天有效期 + 401拦截器友好提示 |

### 7.2 稳定性保障

- **容量评估**：5000 员工规模，单次核算明细 5000 条，分页每页 20 条共 250 页
- **兼容性**：Chrome/Edge/Firefox 最新 3 版本；移动端工资条 iOS Safari / Android Chrome
- **数据安全**：薪资数据不缓存到 localStorage；手机号/身份证号脱敏展示
- **网络异常**：所有 API 请求统一错误拦截，失败自动重试 1 次，仍失败 toast 提示
